## 1) Контекст и проблема
                                                                                                                          
Есть корпус из ~10 длинных монографий (форматы **PDF / TXT / EPUB**). Нужно сделать Telegram‑бота, который:

* отвечает на вопросы пользователей **исключительно опираясь на этот корпус**;
* умеет “бегать по материалам”: сначала искать релевантные фрагменты, затем расширять контекст, затем формировать ответ;
* даёт **цитаты и атрибуцию источников** (какая монография, какой фрагмент/страница/глава).

Тексты монографий будут лежать **в папке в корне проекта** (см. раздел “Контент”).

---

## 2) Цели и критерии успеха

### 2.1 Цели продукта

1. **Правильные ответы с опорой на источники**: бот не “угадывает”, а выводит ответ из корпуса и подкрепляет цитатами.
2. **Удобный UX в Telegram**: быстрый отклик, понятный формат, аккуратный HTML, сообщения не ломаются лимитами Telegram.
3. **Обновляемость базы**: добавили/заменили файл в папке → пересобрали индекс → бот отвечает по новой версии.

### 2.2 Критерии успеха (MVP)

* **≥ 80%** ответов в ручной выборке (например, 50 вопросов) оцениваются как “полезно/точно” человеком‑экспертом.
* **≥ 90%** ответов содержат **2+ цитаты** из корпуса (если вопрос подразумевает фактологию).
* **Среднее время до финального ответа** ≤ 20–30 секунд (при нормальной нагрузке и без деградации модели).
* **0 критических галлюцинаций**: если в корпусе нет ответа, бот явно говорит “в источниках не нашёл” и предлагает 
уточнение.

---

## 3) Не‑цели (чтобы не расползлось)

* OCR (оптическое распознавание) для **сканов PDF** — **вне MVP** (поддерживаем только текстовые PDF).
* Fine‑tuning (дообучение) моделей — **вне MVP**.
* Поддержка “любых документов пользователей” (залили файл — сразу в базу) — **вне MVP**. В MVP источники — только твоя 
папка монографий.
* Полноценная “вики‑БД знаний” с редактором — **вне MVP** (можно добавить позже).

---

## 4) Пользователи и сценарии

### 4.1 Персоны

* **Читатель/исследователь**: задаёт вопросы “что автор понимает под X?”, “как он аргументирует Y?”, “в каких главах 
раскрывается тема Z?”.
* **Админ/редактор** (ты): обновляет корпус, перезапускает индексацию, смотрит статистику и ошибки.

### 4.2 Ключевые User Stories

1. *Как пользователь*, хочу спросить “что автор говорит о …” и получить ответ с цитатами из монографий.
2. *Как пользователь*, хочу уточнить вопрос и чтобы бот учитывал контекст диалога.
3. *Как админ*, хочу положить новые/обновлённые файлы в папку → пересобрать индекс → бот начинает отвечать по новым 
текстам.
4. *Как админ*, хочу видеть статистику: количество запросов, средняя латентность, какие темы спрашивают.

## 5) UX и интерфейс Telegram

### 5.1 Формат ответа

* Ответ в **HTML** (разрешённые теги: `<b>`, `<i>`, `<code>` при необходимости).
* Структура:

  1. **Тезис** (1–2 предложения).
  2. **Разбор** (1–3 смысловых блока).
  3. **Цитаты**: 2–5 цитат с подписью источника.
  4. (Опционально) “Если хочешь — уточни …” (2–3 уточняющих вопроса).

### 5.2 Лимиты Telegram

* Telegram ограничивает длину сообщения (≈4096 символов).
* Бот обязан **резать ответ на части** по ~4000 символов, не разрывая HTML‑теги (аналогично твоему blueprint).

### 5.3 Статус‑сообщения

Во время обработки бот показывает прогресс (редактируя одно “статусное” сообщение):

* `🔍 Ищу в монографиях…`
* `📚 Расширяю контекст…`
* `✍️  Формирую ответ…`

## 6) Контент и управление монографиями

### 6.1 Source of truth

В репозитории есть папка в корне проекта, например:

```
/corpus/
  monograph_01.pdf
  monograph_02.epub
  monograph_03.txt
  ...
```

**Правило MVP:** бот отвечает **только** по документам из `/corpus/`.

### 6.2 Поддерживаемые форматы

* **TXT** — как есть.
* **EPUB** — извлекаем текст из HTML‑разделов.
* **PDF (текстовый)** — извлекаем текст постранично (или с сохранением разрывов страниц).

Если PDF “скан” без текста → документ помечается как “непроиндексирован” (и попадает в отчёт индексации).

### 6.3 Индексация (офлайн)

Индексация запускается отдельно от Telegram‑обработки (важно, чтобы не было “индексация на запросе”).

**Два режима:**

1. `indexer` — CLI/скрипт, который читает `/corpus/`, парсит, чанкует, делает эмбеддинги, пишет в базу.
2. `bot` — веб‑сервер webhook, который только отвечает на вопросы и читает индекс.

## 7) Ключевая продуктовая логика: “агент бегает по материалам”

### 7.1 Подход

Используем агентный цикл в стиле **ReAct (Reasoning + Acting)**: модель сама решает, когда ей нужно вызвать “инструмент 
поиска”, затем получает результаты и строит ответ.

На Yandex Cloud это реализуемо через **function calling (вызовы функций)**: модель не выполняет функции сама, а возвращает
 структурированные параметры вызова, а твой сервис исполняет их и отдаёт результат обратно модели. ([Yandex Cloud][1])

### 7.2 Минимальный набор tools (MVP)

**Tool 1: `search_corpus(query, n_results)`**
Гибридный поиск:

* семантический поиск по векторному индексу (эмбеддинги),
* * keyword‑поиск (полезно для имён, терминов),
* * дедупликация результатов,
* * “расширение” контекста соседними чанками (±N).

**Tool 2: `get_passage(doc_id, chunk_index, radius)`**
Когда модель нашла “центр”, она просит расширенный фрагмент вокруг него.

> В MVP этого достаточно, чтобы агент реально “походил” по текстам: поиск → расширение → ответ.

### 7.3 Поведение агента (правила)

* Перед ответом на фактологические/содержательные вопросы агент **обязан** вызвать `search_corpus` хотя бы 1 раз.
* Максимум tool‑итераций: **4** (конфиг), чтобы не зациклиться.
* Если релевантных фрагментов нет — агент отвечает:

  * что в источниках не нашёл,
  * предлагает 2–3 уточнения,
  * не выдумывает.

## 8) Архитектура на Yandex Cloud

### 8.1 Выбор облачных компонентов

**Вариант MVP (рекомендуемый):**

* **Yandex API Gateway** — публичный HTTPS endpoint для webhook Telegram (серверлесс‑вход).
* **Yandex Serverless Containers** — контейнер с бот‑сервисом (FastAPI/Node).
* **Managed Service for PostgreSQL + pgvector** — хранение чанков, метаданных, векторов и логов запросов. В списке 
поддерживаемых расширений есть `pgvector` (vector similarity search). ([Yandex Cloud][2])
* **Yandex AI Studio**:

  * генерация текста через OpenAI‑совместимый API (совместимость описана в документации; указаны поддержка Responses API и
 др.). ([Yandex Cloud][3])
  * embeddings (эмбеддинги) через REST метод `Embeddings.TextEmbedding`. ([Yandex Cloud][4])
* **Yandex Lockbox** — хранение секретов (токен Telegram, ключи API). (Как паттерн показано в официальном туториале с 
агентом.) ([Yandex Cloud][5])

**Почему так:** минимальная зависимость от сторонних провайдеров, управляемая база, нормальная наблюдаемость.

### 8.2 Telegram webhook на serverless

Yandex Cloud публикует туториал “How to create a Telegram bot with Serverless”, где используется связка API Gateway + 
serverless‑компоненты. ([Yandex Cloud][6])
Мы используем тот же паттерн, только вместо простого hello‑world — RAG/агент‑поток.

## 9) Данные и схема хранения

### 9.1 Таблицы (логическая схема)

**documents**

* `doc_id` (строка, стабильный идентификатор: hash от содержимого/пути)
* `filename`
* `format` (pdf/txt/epub)
* `title` (из метаданных или имени файла)
* `ingested_at`
* `content_hash`
* `status` (indexed / failed / skipped)
* `error` (если failed)

**chunks**

* `chunk_id` (doc_id + chunk_index)
* `doc_id`
* `chunk_index`
* `total_chunks`
* `text`
* `metadata` (json): глава/раздел/страницы/позиция
* `embedding` (vector)
* `tsv` (tsvector) для полнотекстового поиска

**requests**

* `request_id`
* `user_id`
* `username`
* `query`
* `answer`
* `latency_ms`
* `tools_used` (json)
* `created_at`

### 9.2 Чанкинг

Базовая настройка (как в твоём blueprint, но адаптируемо):

* размер чанка: ~1000–1500 символов
* overlap: 150–250 символов
* разрез по предложениям/абзацам (не “по символам в середине фразы”)

### 9.3 Расширение контекста

Для каждого найденного чанка подтягиваем соседние: `chunk_index ± 3…5`, чтобы цитаты были осмысленными, а не “обрывками”.

## 10) Интеграция с моделями (Yandex AI Studio)

### 10.1 Function calling

Модель получает:

* системный промпт,
* запрос пользователя,
* список tools и их JSON‑схемы.

Если модель решает вызвать tool — возвращает structured call; сервис исполняет и возвращает результат в следующий запрос к
 модели. Это поведение описано в документации: модель не исполняет функции сама. ([Yandex Cloud][1])

### 10.2 OpenAI‑совместимость

AI Studio описывает совместимость с OpenAI API (включая Responses API и др.), что позволяет использовать привычные 
SDK‑паттерны (через base_url/project). ([Yandex Cloud][3])

### 10.3 Embeddings API

Эмбеддинги получаем через REST метод `Embeddings.TextEmbedding`. В доках указан endpoint и параметры запроса. ([Yandex 
Cloud][4])

(Технические URL/endpoint — фиксируются в конфиге сервиса и могут лежать в `.env`, в PRD не нормируем “навечно”.)

## 11) Нефункциональные требования

### 11.1 Производительность и латентность

* Webhook должен быстро возвращать `200 OK` Telegram (не держать соединение “вечно”).
* Бот отправляет “статус” сразу, затем — финальный ответ отдельным сообщением.

### 11.2 Надёжность

* Обработка ошибок:

  * LLM недоступна → “временная ошибка, попробуйте позже”
  * База недоступна → аналогично
  * Индекс пуст → бот сообщает, что корпус не проиндексирован

### 11.3 Безопасность

* Секреты (токен Telegram, ключи) — в Lockbox, не в коде/логах. (Паттерн показан в официальном tutorial с агентом.) 
([Yandex Cloud][5])
* Минимальные роли для service account (принцип least privilege).
* Логи не содержат ключей и полных текстов монографий (только chunk_id/метаданные).

### 11.4 Политика “не выдумывать”

* Если ответ не найден в корпусе — бот **обязан** сказать об этом и предложить уточнение.
* Для спорных/неявных мест — бот маркирует вывод как “интерпретация” и показывает цитаты, из которых он следует.

## 12) Аналитика, мониторинг, админ‑функции

### 12.1 Метрики

* `latency_ms` (p50/p95/p99)
* доля ответов с цитатами
* количество tool calls на запрос
* частота “не найдено в источниках”
* ошибки индексации (по документам)

### 12.2 Команды Telegram (MVP)

* `/start` — описание и правила
* `/help` — как спрашивать, примеры
* `/sources` — что за монографии в базе (список titles)
* `/stats` — **только для админа**: количество запросов, средняя латентность, ошибки

### 12.3 Алерты

* пустой ответ
* latency > порога (например, 60–120 секунд)
* рост ошибок модели/базы

## 13) План поставки по фазам (без дат)

### Phase 1 — MVP “вопрос → поиск → ответ с цитатами”

* ingestion из `/corpus/` (pdf/txt/epub)
* индексация в Postgres + pgvector ([Yandex Cloud][2])
* Telegram webhook через API Gateway + Serverless Containers (паттерн из tutorial) ([Yandex Cloud][6])
* агент с 2 tools
* HTML‑ответы + чанкинг сообщений

### Phase 2 — Качество и удобство

* гибридный поиск (вектор + полнотекст)
* персистентная память диалога (Object Storage или Postgres)
* “сильные цитаты”: длинные, не рваные, с чисткой паразитов (`[...]`), как в твоём blueprint
* eval‑набор вопросов и авто‑регрессия качества

### Phase 3 — Knowledge graph (граф знаний)

* отдельная SQLite/Postgres таблица сущностей: термины/персоны/понятия
* tool `find_entity` как быстрый вход для “кто такой X?” (по аналогии с твоей архитектурой)

## 14) Acceptance Criteria (Definition of Done для MVP)

1. **Индексация**

* Запуск `indexer` читает `/corpus/`, создаёт записи в `documents`, `chunks`.
* Для каждого документа статус: indexed/failed.
* Повторный запуск индексатора не дублирует чанки (идемпотентность через `content_hash`).

2. **Поиск**

* `search_corpus` возвращает релевантные чанки + метаданные (doc_title, chapter/page если есть).
* “±N чанков” реально расширяют контекст.

3. **Ответ**

* На типовой вопрос бот даёт ответ с 2–5 цитатами.
* Цитаты имеют подпись источника (название монографии + позиция).
* При отсутствии данных — бот честно сообщает и просит уточнить.

4. **Telegram**

* Нет ошибок HTML разметки.
* Длинные ответы корректно режутся на несколько сообщений.

5. **Безопасность**

* Секреты не хардкодятся, не попадают в логи.

---

## 15) Риски и меры

1. **PDF сложные / ломается извлечение текста**
   → отчёт индексации + пометка failed; OCR отдельной фазой.

2. **Галлюцинации при слабом retrieval**
   → правило: “без найденных фрагментов не отвечать фактологически”; повышаем k, улучшаем гибридный поиск.

3. **Стоимость запросов**
   → лимиты: max tool calls, max tokens, ограничение размера контекста, кэш embeddings для повторяющихся вопросов.

4. **Cold start и скорость**
   → “прогретые инстансы” (если нужно) + быстрый early‑ack в Telegram.
