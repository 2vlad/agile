# Архитектура AI-бота по лекциям (полная спецификация)

Документ описывает архитектуру Telegram-бота с ReAct-агентом, работающего поверх базы лекций. Используй как blueprint для создания аналогичного бота на другом LLM (Яндекс Клауд и т.д.) и других материалах.

---

## Общая схема

```
Пользователь (Telegram)
    ↓
Telegram Bot (polling)
    ├─ текст → agent
    ├─ голос → транскрипция (AssemblyAI) → agent
    └─ файл (.txt/.pdf/.docx) → извлечение текста → agent
    ↓
ReAct Agent (до 8 итераций)
    ├─ LLM решает какой инструмент вызвать
    ├─ find_entity(query) → knowledge.db (SQLite: персоны, фильмы, концепции)
    │   └─ обогащение контекста из ChromaDB
    ├─ search_lectures(query) → ChromaDB (семантический + keyword поиск)
    │   └─ расширение контекста: ±5 чанков вокруг найденного
    └─ LLM формирует ответ с цитатами
    ↓
Telegram: отправка HTML-ответа (чанкинг по 4000 символов)
```

## 1. Подготовка данных (офлайн-пайплайн)

### 1.1 Транскрипция

```
Видео (mp4) → ffmpeg → mp3 (64kbps, 16kHz, mono) → AssemblyAI (язык: ru) → .txt
```

### 1.2 Индексация в векторную базу

**Чанкинг**: разбивка текста на фрагменты ~1000 символов с перекрытием 200 символов. Разбивка по границам предложений (regex `(?<=[.!?])\s+`), не по символам.

**Эмбеддинги**: `paraphrase-multilingual-MiniLM-L12-v2` (мультиязычная модель, поддерживает русский).

**Хранение**: ChromaDB (persistent). Каждый чанк хранится с метаданными:
```json
{
  "id": "{курс}_{имя_лекции}_{индекс_чанка}",
  "metadata": {
    "lecture": "лекция 10.10.2024",
    "course": "1 курс",
    "chunk_index": 5,
    "total_chunks": 42
  }
}
```

ID-схема `{курс}_{лекция}_{индекс}` критична — она позволяет запрашивать соседние чанки для расширения контекста.

### 1.3 Граф знаний (knowledge.db)

SQLite с таблицами:

```sql
directors(id, canonical_name, aliases)           -- режиссёры
persons(id, canonical_name, aliases, roles)      -- философы, писатели, актёры, операторы
films(id, title, year, director_id, aliases)     -- фильмы
concepts(id, name, description, aliases)         -- теории, техники, термины
mentions(id, entity_type, entity_id, lecture, chunk_index, context)  -- сниппеты ~200 символов
```

**Извлечение сущностей**: каждая лекция отправляется в LLM, который извлекает персоны, фильмы, концепции. Для каждой сущности ищутся все упоминания в тексте и сохраняется контекст ~200 символов вокруг.

**Нормализация**: объединение дубликатов ("Годар", "Гадар", "Жан-Люк Годар" → одна запись), перенос не-режиссёров из directors в persons, удаление ложных срабатываний.

## 2. ReAct Agent — ядро AI-архитектуры

### 2.1 Что такое ReAct

Цикл: **Думать → Действовать → Наблюдать → Повторить → Ответить**.

LLM получает вопрос пользователя + набор инструментов. На каждой итерации LLM либо вызывает инструмент (и получает результат), либо формирует финальный ответ. Максимум 8 итераций.

### 2.2 Инструменты агента

**Инструмент 1: `find_entity(query)`**

Назначение: поиск конкретной персоны, фильма или концепции в графе знаний.

```json
{
  "name": "find_entity",
  "description": "Найти персону (режиссёр, философ, писатель, актёр), фильм или концепцию в базе знаний. ИСПОЛЬЗУЙ ПЕРВЫМ при вопросах о конкретном человеке, фильме или термине.",
  "parameters": {
    "query": { "type": "string", "description": "Имя, название фильма или концепции" }
  }
}
```

Логика поиска:
1. Точное совпадение canonical_name
2. Частичное совпадение: `canonical_name LIKE %query%`
3. Совпадение по алиасам: `aliases LIKE %query%`
4. Порядок: directors → persons → films → concepts

Возвращает: имя, алиасы, связанные фильмы, количество упоминаний, до 5 контекстов упоминаний (обогащённых через ChromaDB).

**Fallback**: если сущность не найдена в графе знаний, автоматически вызывается `search_lectures(query, n_results=3)`.

**Инструмент 2: `search_lectures(query, n_results=5)`**

Назначение: гибридный поиск по тексту лекций.

```json
{
  "name": "search_lectures",
  "description": "Гибридный поиск по тексту лекций (семантический + keyword). Для общих вопросов о теории, методах или когда find_entity не нашёл.",
  "parameters": {
    "query": { "type": "string" },
    "n_results": { "type": "integer", "default": 5 }
  }
}
```

Гибридная стратегия поиска:
1. **Семантический**: ChromaDB vector similarity по query
2. **Keyword** (для коротких запросов 1-3 слова): `where_document={"$contains": query}` — точное вхождение текста
3. Результаты объединяются, дубликаты удаляются, ограничиваются `n_results`

**Расширение контекста**: для каждого найденного чанка подгружаются ±5 соседних чанков (по ID-схеме `{курс}_{лекция}_{индекс ± N}`). Это даёт ~30-40 предложений контекста вокруг каждого найденного фрагмента.

### 2.3 Обогащение контекста упоминаний

Функция `enrich_mention_context()`:

```
knowledge.db mention: короткий сниппет ~200 символов
    ↓
ChromaDB query: найти ближайший чанк в той же лекции
    ↓
Извлечь chunk_index, course из метаданных
    ↓
get_context_chunks(): загрузить ±3 чанка по ID
    ↓
Результат: ~3000 символов развёрнутого контекста
```

### 2.4 Цикл работы агента

```python
def run_agent(user_query, user_id, max_iterations=8):
    messages = [system_prompt] + conversation_history + [user_query]

    for i in range(max_iterations):
        response = llm.chat(messages, tools=TOOLS, tool_choice="auto",
                           max_tokens=3000, temperature=0.3)

        if response.has_tool_calls:
            for tool_call in response.tool_calls:
                result = execute_tool(tool_call.name, tool_call.args)
                result = truncate(result, max_chars=12000)
                messages.append(tool_result(tool_call.id, result))
        else:
            # Финальный ответ
            return clean_html(response.content)

    # Если 8 итераций исчерпаны — принудительно запросить ответ
    messages.append("Сформируй ответ на основе собранной информации")
    return llm.chat(messages, max_tokens=4000).content
```

### 2.5 Системный промпт (ключевые директивы)

Промпт генерируется динамически в `get_system_prompt()`. Ключевые правила:

**Стратегия поиска**:
- ВСЕГДА ИЩИ СНАЧАЛА — никогда не говори "не могу помочь" без поиска
- ШИРОКИЙ вопрос (напр. "расскажи про Годара") → 2-3 вызова инструментов (find_entity + search_lectures с вариациями)
- УЗКИЙ/уточняющий вопрос → 1 точный вызов

**Формат цитат**:
- Полные, длинные цитаты (не обрезать на полуслове)
- Формат: `<i>«цитата»</i> (лекция за ДД месяц ГГГГ)`
- Убирать слова-паразиты (ну, вот, как бы, там, типа) с пометкой `[...]`

**Структура ответа**:
- Краткий тезис (1-2 предложения)
- Развитие с цитатами из РАЗНЫХ лекций
- Группировка по смысловым блокам
- Целевой объём: 800-2000 символов

**Стиль**:
- Разговорный, без пафоса
- НЕ "шедевр", "гений", "произведения" → "фильмы"
- Говорить на равных, без благоговения
- Только HTML-теги (`<b>`, `<i>`), НЕ markdown

## 3. Память диалогов

**Хранение**: in-memory dict `{user_id: {messages, last_activity, chat_id}}`

**TTL**: 1 час неактивности → очистка + проактивное уведомление пользователю

**Суммаризация**: при накоплении 10 сообщений:
1. Первые 6 сообщений → суммаризация через LLM → "Контекст: краткое содержание"
2. Последние 4 сообщения — без изменений
3. Итого: ~8 сообщений в памяти

**Тримминг контекста** (если общий размер превышает лимит):
1. Мягкий: обрезка tool-результатов (кроме 2 последних) до 2000 символов
2. Жёсткий: удаление старых tool-результатов (кроме 2 последних)

## 4. Telegram Bot

### Обработка сообщений

- **Текст**: напрямую в агент
- **Голос**: скачать файл → AssemblyAI (язык: ru) → текст → агент
- **Файл** (.txt/.pdf/.docx, макс 8000 символов): извлечь текст → агент с промптом "Проанализируй"

### HTML-чанкинг ответов

Телеграм лимит: 4096 символов. Ответы разбиваются по 4000 символов с учётом:
- Приоритет разбивки: `\n\n` (параграфы) → `\n` (строки)
- Не разрывать HTML-теги

### Статус-сообщения

Агент передаёт callback `on_status()`, который обновляет сообщение в реальном времени:
```
🔍 Анализирую вопрос...
🎥 Ищу в базе знаний: Годар
📚 Ищу лекции: монтаж Годара
💬 Формирую ответ...
```

### Мониторинг

**Аномалии** (с кулдауном 5 минут):
- Ответ начинается с "Ошибка" → alert
- Пустой ответ → alert
- Латентность > 120 секунд → alert

Алерт отправляется админу в Telegram.

### Статистика (stats.db)

```sql
requests(id, user_id, username, query, response, response_time_ms, tools_used, timestamp)
```

Команда `/stats [дней]` показывает: запросов, уникальных пользователей, среднее время ответа, топ пользователей, последние запросы.

## 5. Параметры и лимиты

| Параметр | Значение | Назначение |
|----------|----------|------------|
| Модель LLM | moonshotai/kimi-k2.5 | Основная генерация (через OpenRouter) |
| Контекст модели | 262,144 токенов | Лимит контекстного окна |
| max_tokens (основной) | 3,000 | Генерация на каждой итерации |
| max_tokens (финальный) | 4,000 | После исчерпания итераций |
| temperature | 0.3 | Детерминированные ответы |
| Макс итераций | 8 | Предотвращение зацикливания |
| Результатов поиска | 5 | На один вызов search_lectures |
| Контекст чанков | ±5 | Вокруг каждого результата поиска |
| Макс символов tool result | 12,000 | Обрезка результатов инструментов |
| TTL диалога | 3,600 сек | Автоочистка через 1 час |
| Суммаризация после | 10 сообщений | Сжатие истории диалога |
| Чанк лекции | ~1,000 символов | С перекрытием 200 |
| Эмбеддинг модель | paraphrase-multilingual-MiniLM-L12-v2 | Мультиязычная |

## 6. Что адаптировать для нового бота

### Замена LLM
- OpenRouter + Kimi K2.5 → Yandex Cloud GPT или другой LLM
- Ключевое: LLM должен поддерживать **tool use / function calling** (нативный, не через парсинг текста)
- Параметры `max_tokens`, `temperature`, контекстное окно — подстроить под выбранную модель

### Замена эмбеддингов
- `paraphrase-multilingual-MiniLM-L12-v2` — можно оставить (open source, работает локально)
- Или заменить на эмбеддинги Yandex Cloud (если нужна лучшая поддержка русского)

### Замена материалов
- Подготовить транскрипции/тексты в .txt
- Адаптировать `indexer.py` под новые источники (папки, форматы имён)
- Адаптировать `extract_entities.py` под новую предметную область (вместо режиссёров/фильмов — свои типы сущностей)
- Изменить системный промпт под новую тему и стиль ответов

### Что оставить без изменений
- Архитектура ReAct-агента (цикл, два инструмента)
- Гибридный поиск (семантический + keyword)
- Расширение контекста через соседние чанки
- Обогащение упоминаний из графа знаний через векторную базу
- Память диалогов с суммаризацией
- HTML-чанкинг для Telegram
- Мониторинг аномалий и статистика
